# BUG-047: Bot demo-moda não responde

**Data:** 2026-03-18  
**Estado:** Resolvido

## Sintoma

- Webhook Evolution configurado: `http://10.11.16.112:3000/webhook/demo-moda`, enabled, event `MESSAGES_UPSERT`.
- Logs do engine mostravam **zero** actividade para demo-moda — nenhum request chegava.

## Root cause

1. **Rota inexistente:** O engine registava apenas `POST /webhook` e `POST /webhook/messages`. A Evolution enviava para `POST /webhook/demo-moda`, que em Express **não correspondia** a nenhuma rota → **404**. O body nunca chegava ao `webhookRouter`.
2. **IP incorrecto:** O IP do serviço Docker no servidor (Virtual IP do overlay) era **10.11.10.116**, não 10.11.16.112. Assim, mesmo que a rota existisse, a Evolution poderia estar a enviar para um host/porta errados.
3. **Config demo-moda:** O handler espera `identity`, `supervisors`, `systemMessages` e usa `config.stock` em várias funções; clientes sem Sheets (ex.: demo-moda) não tinham `stock` e algumas funções (ex.: `getClienteByTelefone`) podiam falhar se `stockConfig` fosse `null`.

## Fix

1. **Nova rota:** Adicionada `POST /webhook/:instanceName` em `index.js`, usando o mesmo `webhookRouter`. O middleware `webhook-router.js` passou a considerar `req.params.instanceName` na resolução da instância: `body.instance || body.instanceName || ... || req.params.instanceName`.
2. **Webhook Evolution:** URL actualizada para o IP correcto do serviço: `http://10.11.10.116:3000/webhook/demo-moda` (via `POST /webhook/set/demo-moda` na Evolution API).
3. **Config demo-moda:** Em `config/clientes/demo-moda.js` foram adicionados `identity`, `supervisors`, `systemMessages`, `evolutionInstance`, `stock: null`, `payment: null`, `products: null` para compatibilidade com o handler multi-tenant.
4. **Protecção sem stock:** Em `src/integrations/google-sheets.js`, `getClienteByTelefone` e `getPerfisExpirados` passam a retornar cedo quando `!stockConfig || !stockConfig.sheetName`, evitando erros em clientes sem planilha de stock.

## Prevenção — Checklist para novos bots

Antes de considerar um novo bot (nova instância Evolution) operacional:

1. **Config do cliente:** Existe ficheiro em `config/clientes/<slug>.js` (ou entrada equivalente) com pelo menos: `identity`, `supervisors`, `evolutionInstance`/`slug`, e `stock`/`payment`/`products` conforme o tipo de negócio (ou `null` se não aplicável).
2. **Registro no engine:** A instância está registada no registry (via `src/router.js` ou carregamento dinâmico) com `evolutionInstance` igual ao nome da instância na Evolution.
3. **Rota:** O engine expõe `POST /webhook/:instanceName` (e/ou aceita `instance` no body em `POST /webhook`). O URL do webhook na Evolution deve ser exactamente esse path (ex.: `http://<HOST>:3000/webhook/<instanceName>`).
4. **IP/host do webhook:** O URL do webhook na Evolution deve apontar para o **IP ou hostname real** do serviço que corre o engine (ex.: Virtual IP do Docker overlay no servidor). Confirmar com:
   ```bash
   ssh root@<SERVER> "docker service inspect <SERVICE_NAME> --format '{{range .Endpoint.VirtualIPs}}{{.Addr}}{{end}}'"
   ```
5. **Teste de payload:** Simular um `POST` para o URL do webhook com um payload mínimo (evento `MESSAGES_UPSERT`, `instance`, `data.key`, `data.message`) e confirmar 200 e que os logs do engine mostram a mensagem recebida para essa instância.

## Referências

- Evolution API: `POST /webhook/set/{instance}` para configurar webhook; evento `MESSAGES_UPSERT`.
- Remetente nas mensagens: usar sempre `data.key.remoteJid`; nunca `req.body.sender`.
